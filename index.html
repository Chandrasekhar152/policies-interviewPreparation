Q6What is a REST API? Does your project use it?▼
REST API is how the frontend and backend talk to each other.
Think of it like ordering food. You tell the waiter what you want. The waiter brings it from the kitchen.
Here — React is the customer. FastAPI is the waiter. Database is the kitchen.
GET means "give me some data." No change happens.
POST means "save this new data."
PUT means "update this existing data."
Yes. My project uses REST API everywhere.
Q7What is CORS? Why did you use it?▼
CORS is a browser safety rule.
By default, a browser blocks two different addresses from talking to each other.
My React runs at localhost:3000. My FastAPI runs at localhost:8000. These are two different addresses.
So the browser was blocking them.
I added CORS settings in FastAPI. I said — allow requests from localhost:3000.
After that, they can talk to each other without any problem.

Backend — FastAPI & Python
12 Q
Q13Why did you use FastAPI and not Flask or Django?▼
FastAPI was mentioned in the document. So I used it.
But I also understand why it is good.
FastAPI has a testing page at /docs. I can test all my APIs there — no extra work.
FastAPI checks data automatically. If wrong data comes in, it gives a clear error.
Flask is simple but has fewer features. Django is too big for this project.
FastAPI was the right fit.
Q14What is a router in FastAPI?▼
A router splits the code into separate files.
I did not put all APIs in one big file. That would be messy.
I made separate files. claims.py for claims. admin.py for admin. policies.py for policies.
Each file has its own router.
In main.py, I join all routers together.
Now the code is clean. Easy to find. Easy to fix.
Q15What is Pydantic? Why did you use it?▼
Pydantic checks if the data sent by the user is correct.
Example: when filing a claim, the user must send amount, type, and date.
If they forget one — Pydantic gives an error message automatically.
If they send wrong type — like a word instead of a number — Pydantic catches it.
I wrote these rules in schemas.py.
FastAPI uses them automatically. I do not have to write checking code in every API.
Q16What is Celery? Why did you use it?▼
Celery runs tasks in the background.
Sending an email takes a few seconds.
If I send the email inside the API — the admin has to wait. That feels slow.
With Celery — the API replies to admin immediately. Done!
Then Celery sends the email quietly in the background.
The admin does not wait. The app feels fast.
Celery was mentioned in the project document. I built it exactly as described.
Q17How does fraud detection work in your project?▼
When a user files a claim, 4 checks run automatically.
Rule 1 If the amount is more than ₹1 lakh — flag it. High risk.
Rule 2 If the claim is filed within 7 days of buying the policy — flag it. Too fast.
Rule 3 If the same amount was claimed before for the same policy — flag it. Looks like a copy.
Rule 4 If 2 or more claims are filed within 3 days — flag it. High risk.
If a high risk flag is found — the claim status changes to "under review" automatically.
All flags are saved in the database. Admin can see them.
Q18How does the recommendation engine work?▼
The recommendation engine gives a score to each policy for the user.
Higher score = better match for that user.
It looks at the user's age, income, family size, and health condition.
+25 points If the policy price is less than 5% of yearly income. It is affordable.
+20 points If the user is high risk and the policy has low deductible. Good coverage.
+10 points If the user has 2 or more family members.
Top 5 policies by score are shown to the user with a reason.
Q19What is SQLAlchemy? How did you use it?▼
SQLAlchemy lets me use Python to work with the database.
I do not have to write SQL commands directly.
I wrote Python classes in models.py. Each class is one database table.
To save a new claim — I write db.add(new_claim). SQLAlchemy does the rest.
It is cleaner. It is safer. It is easier to read.
Q20What is dependency injection in FastAPI?▼
Dependency injection means FastAPI gives the function what it needs automatically.
Example 1 I write db = Depends(get_db). FastAPI opens the database connection for me. I do not have to open or close it myself.
Example 2 I write current_user = Depends(get_current_user). FastAPI reads the token and gives me the user. I do not write this in every API.
It removes repeated code. It keeps things clean.
Q21What does db.commit() and db.rollback() do?▼
Think of the database like a notebook.
When you write something — it is in pencil first. Not saved yet.
db.commit() = Press Save. Changes are permanent now.
db.rollback() = Press Undo. All unsaved changes are cancelled.
In my upload code — if Google Drive upload fails — I call rollback.
This means no broken data is saved. The database stays correct.
Q22How did you send emails in your project?▼
Email notifications were part of the document — Week 6.
I used Python's smtplib library to send emails through Gmail.
When admin changes a claim status — Celery runs in the background.
It finds the user's email from the database.
It sends an email — "Your claim status is now [new status]."
I used a Gmail App Password — a special password only for apps. Safer than the real password.
Q23How did you handle errors in your API?▼
When something goes wrong — I use HTTPException to send an error back.
404 = Not found. Example — claim ID does not exist.
403 = Not allowed. Example — user trying to open admin page.
500 = Something broke on the server.
The frontend reads this error code and shows the right message to the user.
Q24What is the difference between GET, POST, and PUT?▼
These are ways the frontend sends requests to the backend.
GET = "Give me data." Nothing changes. Example — show all my claims.
POST = "Save new data." Example — file a new claim.
PUT = "Update existing data." Example — admin changes claim status.
We always use the right method for the right action. That is the REST rule.

Database — PostgreSQL
10 Q
Q25How many tables are in your database? What does each one store?▼
The database schema was given in the project document. There are 9 tables.
Users Name, email, password, age, and risk profile of each user.
Providers Insurance company name and country.
Policies Policy type, price, coverage, and deductible.
UserPolicies Which user bought which policy. Start and end date.
Claims Each claim — amount, date, and current status.
ClaimDocuments Google Drive link of uploaded documents.
Recommendations Score and reason for each suggested policy.
FraudFlags Which fraud rule was broken and how serious it is.
AdminLogs What actions the admin has done.
Q26What is a foreign key? Give an example.▼
A foreign key connects one table to another table.
Think of Students list and Marks list. Each mark has a Student ID. That connects the mark to the student.
That Student ID is a foreign key.
In my project — Claims table has user_policy_id. This connects each claim to one user policy.
You cannot file a claim without a valid policy. The foreign key enforces that.
Q27What is JSONB? Why did you use it?▼
JSONB stores flexible data inside one database column.
Like a small dictionary — different keys for different rows.
risk_profile Each user has different details — age, income, health, family size. These are not the same for everyone. So I store it all in one JSONB column.
coverage Each policy type covers different things. Health covers hospital. Travel covers flights. JSONB handles this flexibility.
Both were given in the schema in the project document.
Q28What is the difference between primary key and unique key?▼
Both stop duplicate values. But they are different.
Primary Key Every table has exactly one. It identifies each row. It cannot be empty. In my tables — the id column is the primary key.
Unique Key Stops duplicates in a column. But it is not the main identifier. Example — email in Users. No two users can have the same email.
Q29What is CASCADE delete? Where did you use it?▼
CASCADE delete means — if a parent row is deleted, child rows are also deleted automatically.
Simple example — delete a student, their marks are also deleted.
Without this — you get leftover marks with no student. That is bad data.
In my project — if a UserPolicy is deleted, all its Claims are also deleted.
If a Claim is deleted, its FraudFlags and ClaimDocuments are also deleted.
The database stays clean. No leftover data.
Q30What is an ORM? Why is it better than writing SQL?▼
ORM lets me use Python to talk to the database. No SQL needed.
I used SQLAlchemy as my ORM.
Reason 1 Code is easier to read. Python is cleaner than SQL mixed in strings.
Reason 2 It is safer. It protects from hacker tricks like SQL injection automatically.
Reason 3 If I switch databases later — very little code changes needed.
Q31What is a JOIN? Did you use it?▼
JOIN gets data from two connected tables at the same time.
Example — you want a student's name AND their marks. Name is in one table. Marks in another. JOIN combines them.
Yes. I used JOIN in my project.
To get claims for the current user — I joined Claims with UserPolicies.
Claims table does not have user_id directly. UserPolicies has it. So I JOIN them to filter by user.
Q32Why did you use server_default=func.now() for timestamps?▼
This tells the database — "fill in the current time automatically when a row is saved."
I do not have to send the time from Python.
The database server's own clock is used. It is always accurate and consistent.
I used this in Claims, ClaimDocuments, FraudFlags, and AdminLogs.
Q33Why did you use NUMERIC instead of float for money?▼
Money needs exact values. Example — ₹5,250.75 exactly.
Float type sometimes stores ₹5,250.74999 because of how computers handle decimals.
That small error looks tiny. But when you add money for thousands of users — the errors add up.
NUMERIC(12, 2) stores the exact value. Always 2 decimal places. No rounding error.
This is the standard way to store money in any database.
Q34What does the Recommendations table store?▼
When a user asks for recommendations — the engine scores all policies.
Each score and reason is saved in the Recommendations table.
Before saving new ones — old ones are deleted. Data stays fresh.
Top 5 by score are shown to the user.
Admin can also check this table to see what was suggested to each user.

Login & Security — JWT
8 Q
Q35What is JWT? How does it work in your project?▼
JWT was mentioned in the project document. It stands for JSON Web Token.
It is a way to verify who the user is. Without checking the database every time.
Think of it like a concert pass. Once you have the pass — the guard just checks the pass. No need to call the ticket office each time.
Step 1 User logs in with email and password.
Step 2 Backend checks. If correct — it creates a JWT token and sends it.
Step 3 Frontend saves this token. Sends it with every future request.
Step 4 Backend reads the token, verifies it, and knows who the user is.
Q36What are the 3 parts of a JWT token?▼
A JWT token looks like this — aaaaa.bbbbb.ccccc
Three parts. Each separated by a dot.
Part 1 Header Says which method was used. I used HS256.
Part 2 Payload Has user info — email and expiry time. Readable but cannot be changed without being caught.
Part 3 Signature Made using a secret key only my server knows. If anyone changes the token — signature breaks. Backend rejects it.
Q37How did you store passwords safely?▼
I never stored the real password in the database.
I stored a hashed version.
Hashing converts the password into a scrambled string. It cannot be reversed.
Example — "hello123" becomes "x9q$2mK#r..." — no one can go back to "hello123".
I used bcrypt for hashing. It is very safe and popular.
When user logs in — I hash what they typed and compare with the stored hash. If they match — login allowed.
Q38When does the JWT token expire? What happens after?▼
The token expires after 30 minutes.
I set this in the code — ACCESS_TOKEN_EXPIRE_MINUTES = 30.
After 30 minutes — if the user tries anything — the backend rejects it.
It sends a 401 error. That means "you are not logged in."
The user has to log in again to get a new token.
The document mentioned refresh tokens too. That is the next improvement I would add.
Q39What is OAuth2? How did you use it?▼
OAuth2 lets one app get permission to use another service. Without sharing your password.
I used it in 2 places.
For Login FastAPI's OAuth2PasswordBearer reads the JWT token from each request. Gives it to my code.
For Google Drive First time — browser opens. Asks "allow this app to use your Drive?" After yes — credentials are saved in token.pickle. No more browser popups after that.
Q40How did you handle admin access?▼
I made a function called admin_only.
It checks — is the logged-in user's email the same as the admin email?
If yes — allowed. If no — 403 error. "Admin access required."
All admin APIs use this check.
I know checking by hardcoded email is not ideal for a big app. A better way is to store a role in the database. That is what I would improve next.
Q41What is the difference between authentication and authorization?▼
These two sound similar. But they are different.
Authentication = "Who are you?" — Checking email and password at login. You get a JWT token after this.
Authorization = "What can you do?" — Even after login, not everyone can open the admin page. Only admin is allowed.
Easy way to remember it — authentication is about identity. Authorization is about permission.
Q42Are there any security issues in your project?▼
Yes. I will be honest.
The JWT secret key is written in the code directly. It should be in a private environment file.
The Gmail password is also in the code. Same problem.
Admin is checked by hardcoded email. Should be a role in the database.
These are fine for an internship learning project. But not for a real production app.
I am aware of these. I know how to fix them. That is what I would do next.

Google Drive & File Uploads
6 Q
Q43The document says AWS S3. Why did you use Google Drive?▼
Yes. The project document from Infosys Springboard mentioned AWS S3.
I made only this one change — I used Google Drive instead.
Reason — AWS S3 needs a credit card. It has a cost. I am a student.
Google Drive is completely free. It has 15 GB. Enough for a development project.
The logic is exactly the same — upload file, get a link, save link in database.
I achieved the same goal. This is the only change from the document.
Q44What happens step by step when a user uploads a file?▼
Step 1 User picks a file and clicks Upload.
Step 2 React sends the file to FastAPI.
Step 3 FastAPI checks — does this claim belong to this user? If no — 404 error.
Step 4 File is saved temporarily on the server.
Step 5 My code uploads this temp file to Google Drive.
Step 6 Google Drive gives back a file ID. My code makes it public and creates a shareable link.
Step 7 This link is saved in the ClaimDocuments table in the database.
Step 8 Temp file on the server is deleted. Always — even if something went wrong.
Step 9 The link is sent back to React as the response.
Q45What is token.pickle?▼
token.pickle is a file that saves the Google login credentials.
First time the app runs — browser opens. Google asks "allow this app?"
After you click Yes — Google gives credentials. My code saves them to token.pickle.
Next time — the app reads from this file. No browser popup again.
If credentials expire — it auto-refreshes. No manual step needed.
User only gives permission once. Everything after is automatic.
Q46Why did you use a finally block in the upload code?▼
The finally block runs always — whether the upload worked or failed.
When a user uploads a file — I save it as a temp file on the server first.
After uploading to Google Drive — I must delete this temp file.
If I forget to delete — the server fills up with leftover files slowly.
I put the delete code inside finally. So it always runs. No matter what.
Q47What Google Drive permission did you use? Why?▼
I used the scope drive.file.
This means — my app can only see files that it uploaded itself.
It cannot see the user's personal photos or documents.
This is safer. My app only gets the minimum permission it needs.
If I used full Drive access — that would be a privacy risk. I avoided that.
Q48What does resumable=True mean in your upload code?▼
Resumable means — if internet breaks during upload — it continues from where it stopped.
Without this — if internet breaks at 70% — file upload starts from 0% again.
With resumable=True — it continues from 70%.
Claim documents are sometimes large scanned PDFs.
Resumable makes the upload more reliable for large files.

Q50What is React Router?▼
React Router lets the user go between pages without full browser reload.
Normal website — you click a link — browser downloads a whole new page. Slow.
With React Router — React just shows a different component. Very fast. No reload.
In my App.js — I defined the routes. /login shows Login page. /policies shows Policies page. And so on.
Q51How does the frontend talk to the backend?▼
The frontend uses the fetch function to send HTTP requests to the backend.
Example — My Claims page opens. React calls fetch("http://localhost:8000/claims/").
Before sending — it adds the JWT token to the request. Like an ID card.
Backend receives the request. Finds the claims for that user. Sends back the list.
React saves the list and shows it on the screen.
Q52What is the difference between state and props?▼
Both store data. But they are different.
State Data that belongs to this component. It can change. When it changes — screen updates automatically. Example — the list of claims I fetched from the backend.
Props Data given to this component by its parent. Cannot be changed. Just read. Example — parent page gives one claim to a ClaimCard component.
Simple rule — State is your own. Props is given by someone else.
Q53What is useEffect? Where did you use it?▼
useEffect runs code when the page first loads.
I used it on every page that needs data from the backend.
Example — Policies page opens. useEffect runs. It calls GET /policies API. Gets the list. Saves in state. React shows the policies on screen.
Without useEffect — the page opens empty. No data. Because React does not know when to fetch.
Q54What does the Admin Dashboard show?▼
Only the admin can see this page.
Summary Total users, claims, fraud flags, premium collected, amount claimed.
All Claims Full list with documents and fraud flags for each.
Update Status Admin can change claim status. User gets email automatically via Celery.
Export CSV Download all claims as a spreadsheet file.
Q55What is a component in React?▼
A component is one small piece of a web page.
Think of a web page like a house made of Lego blocks. Each block is a component.
Login page is one component. Policies page is another.
Each component has its own code and its own data.
Big benefit — I can reuse a component. Write once. Use in many pages.
